org.jsimpledb

Class JSimpleDB

  • java.lang.Object
    • org.jsimpledb.JSimpleDB

  • public class JSimpleDB
extends Object
    A JSimpleDB database.

    JSimpleDB consists of two main layers. The lower level "core" API provides a rigourous database abstraction on top of a key/value store. It supports simple fields of any atomic Java type, as well as list, set, and map complex fields, tightly controlled schema versioning, simple and composite indexes, and lifecycle and change notifications. The core API is provided via the Database class.

    This JSimpleDB class represents the upper layer. It sits on top of the core Database class and provides a fully type-safe Java view of a Database, where all access is through user-supplied Java model classes. In addition, a JSimpleDB provides automatic incremental JSR 303 validation. Database types and fields, as well as listener methods, are all declared using various Java annotations.

    User-provided Java model classes are typically abstract and declare fields as abstract Java bean methods. In any case, JSimpleDB generates concrete subclasses at runtime; these runtime classes will implement the JObject interface. The corresponding Java model objects are stateless; all database field state is derived from whichever transaction is associated with the current thread.

    Not counting "snapshot" objects (see below), this class guarantees that for each ObjId there will only ever be a single, globally unique, JObject Java model object. Therefore, the same Java model objects can be used in and out of any transaction.

    "Snapshot" objects are in-memory copies of regular database objects that may be imported/exported to/from transactions. Snapshot JObjects are distinct from regular database JObjects; their state is contained in an in-memory SnapshotJTransaction. See JObject.copyOut and JObject.copyIn.

    See Also:
    JObject, JTransaction, JSimpleDBFactory, org.jsimpledb.annotation

      • getActualVersion

        public int getActualVersion()
        Get the schema version that this instance used for the most recently created transaction.

        If no transactions have been created yet, this returns zero. Otherwise, it returns the schema version used by the most recently created transaction.

        If the version passed to the constructor was zero, this method can be used to read the highest schema version seen in the database by the most recently created transaction.

        If the version passed to the constructor was non-zero, and at least one transaction has been created, this method will return the same value.

        Returns:
        the schema version that this instance used in the most recently created transaction

      • getJObject

        public JObject getJObject(ObjId id)
        Get the Java model object that is has the given ID. This returns the object associated with normal (i.e., non-snapshot) JTransactions. To get the object associated with a snapshot transaction, invoke JTransaction.getJObject() on the snapshot transaction.

        This method guarantees that for any particular id, the same Java instance will always be returned. Note: while for any ObjId there is only one globally unique JObject per JSimpleDB shared by all JTransactions (the object returned by this method), each SnapshotJTransaction maintains its own distinct pool of unique "snapshot" JObjects.

        A non-null object is always returned, but the corresponding object may not exist in a given transaction. In that case, attempts to access its fields will throw DeletedObjectException. Also, it's possible that id corresponds to an object type which no longer exists in the schema version associated with this instance. In that case, an UntypedJObject is returned.

        Parameters:
        id - object ID
        Returns:
        Java model object
        Throws:
        IllegalArgumentException - if id is null
        See Also:
        JTransaction.getJObject()
